## 一、什么是工具系统？（小白通俗解释）

### 1.1 生活化类比

想象你有一个**超级智能助手**（比如 AI），它很聪明，但**只能说话**，不能动手做事。

**工具系统**就是给这个智能助手配备各种"工具"，让它能**真正动手干活**：

| 工具 | 类比 | 能做什么 |
|------|------|---------|
| `read` (读取文件) | 📖 眼睛 | 读取电脑上的文件内容 |
| `write` (写入文件) | ✏️ 笔 | 在电脑上写文件、保存内容 |
| `exec` (执行命令) | 🔨 锤子 | 执行系统命令（如列出文件） |

**没有工具的 AI**：像一个只能说话不能动的机器人
**有工具的 AI**：像一个有手有脚、能读能写的全能助手！

### 1.2 在赛博小镇中的作用

在赛博小镇中：
- 角色（作家、医生等）需要**读写文件**来保存日记、记录
- 需要**执行命令**来查询信息
- 工具系统让这些角色能够**与电脑交互**

## 三、核心概念详解

### 3.1 AgentTool 接口（工具的标准规范）

这是所有工具必须遵守的**标准格式**，就像产品必须符合质量标准一样。

```typescript
// 位置: src/AI/tools/core.ts
interface AgentTool<TParameters = any, TResult = unknown> {
  name: string;                    // ★ 工具名称（唯一标识）
  label?: string;                  // 显示名称（给人看的）
  description: string;             // ★ 描述（给AI看，告诉它能做什么）
  parameters: z.ZodType<TParameters>; // ★ 参数定义（用Zod验证）
  ownerOnly?: boolean;             // 是否仅限所有者使用
  displaySummary?: string;         // 显示摘要
  
  execute: (                       // ★ 核心方法：执行工具
    toolCallId: string,
    params: TParameters,
    signal?: AbortSignal,
    onUpdate?: (progress: string) => void
  ) => Promise<AgentToolResult<TResult>>;
}
```

**逐行解释**：

```typescript
// 示例：一个简化版的"读取文件"工具
const myTool: AgentTool = {
  name: 'read',                    // 工具名字叫 "read"
  label: '读取文件',               // 显示名：读取文件
  description: '读取指定路径的文件', // AI看到这个描述就知道什么时候用它
  parameters: z.object({           // 定义参数格式
    file_path: z.string()          // 需要一个字符串类型的 file_path 参数
  }),
  
  async execute(toolCallId, params, signal, onUpdate) {
    // 这里是实际执行的代码
    const content = await fs.readFile(params.file_path);
    return { content: [{ type: 'text', text: content }] };
  }
};
```

### 3.2 AgentToolResult（统一的返回格式）

无论什么工具，返回结果都要用这个格式：

```typescript
interface AgentToolResult<T = unknown> {
  content: Array<{
    type: 'text' | 'image';        // 内容类型：文字或图片
    text?: string;                 // 文字内容
    source?: {                     // 图片数据（base64编码）
      type: 'base64';
      media_type: string;
      data: string;
    };
  }>;
  details?: T;                     // 结构化的详细信息（可选）
}
```

**示例返回值**：

```typescript
// 返回文本结果
{
  content: [{ 
    type: 'text', 
    text: '这是文件的内容...' 
  }],
  details: { file_path: '/a/b.txt', size: 100 }
}

// 返回图片结果
{
  content: [{
    type: 'image',
    source: {
      type: 'base64',
      media_type: 'image/png',
      data: 'iVBORw0KGgo...'  // base64编码的图片数据
    }
  }]
}
```

### 3.3 ToolExecuteArgs（执行参数）

```typescript
interface ToolExecuteArgs {
  toolCallId: string;      // 本次调用的唯一ID（用于追踪）
  params: unknown;         // LLM传入的参数
  signal?: AbortSignal;    // 中止信号（可以取消长时间运行的工具）
  onUpdate?: (progress: string) => void;  // 进度回调函数
}
```

## 四、内置工具详解

### 4.1 文件读取工具 (`createReadTool`)

**位置**: `src/AI/tools/fs-tools.ts`

**功能**: 读取电脑上的文件内容

```typescript
// 创建读取工具
const readTool = createReadTool({ workspaceRoot: '/my/project' });

// 使用方式
const result = await readTool.execute('call_001', { file_path: 'README.md' });
console.log(result.content[0].text);  // 输出文件内容
```

**安全特性**：
- ✅ **路径保护**: 只能读取工作区内的文件，防止读取敏感系统文件
- ❌ 拒绝访问 `/etc/passwd` 等系统文件

```typescript
// 安全检查逻辑
const fullPath = path.resolve(workspaceRoot, params.file_path);
if (!fullPath.startsWith(workspaceRoot)) {
  throw new Error(`拒绝访问：路径超出工作区范围`);
}
```

### 4.2 文件写入工具 (`createWriteTool`)

**功能**: 向电脑写入/保存文件

```typescript
const writeTool = createWriteTool({ workspaceRoot: '/my/project' });

await writeTool.execute('call_002', {
  file_path: 'note.txt',
  content: 'Hello World!'
});
// → 返回: { content: [{ text: '文件已写入：/my/project/note.txt' }] }
```

**特殊属性**：
- `ownerOnly: true` — 只有所有者（管理员）可以使用
- 同样有路径保护

### 4.3 命令执行工具 (`createExecTool`)

**功能**: 在终端执行 shell 命令

```typescript
const execTool = createExecTool({
  allowedCommands: ['ls', 'cat', 'pwd'],  // 白名单命令
  timeout: 30000                           // 超时30秒
});

const result = await execTool.execute('call_003', { command: 'ls -la' });
// → 返回命令输出结果
```

**安全机制**：
- ✅ **命令白名单**: 只允许指定的命令
- ✅ **超时控制**: 防止命令无限运行
- ⚠️ `ownerOnly: true` — 默认仅管理员可用

## 五、ToolRegistry（工具注册表）⭐ 核心类

### 5.1 是什么？

`ToolRegistry` 是工具系统的**大管家**，负责：
- **注册**新工具
- **查找**工具
- **按策略过滤**工具（控制谁能用什么工具）

### 5.2 基本使用

```typescript
import { ToolRegistry, createDefaultToolRegistry } from './tools';

// 方法1: 创建默认注册表（预装了 read/write/exec）
const registry = createDefaultToolRegistry();

// 方法2: 手动创建并注册
const registry = new ToolRegistry({ isOwner: true });
registry.registerTool(createReadTool());
registry.registerTool(createWriteTool());
registry.registerTool(createExecTool());

// 获取所有工具
const allTools = registry.getAllTools();
console.log(`共 ${allTools.length} 个工具`);

// 获取特定工具
const readTool = registry.getTool('read');

// 获取给LLM用的工具定义
const definitions = registry.getToolDefinitions();
```

### 5.3 API 完整列表

| 方法 | 说明 | 参数 | 返回值 |
|------|------|------|--------|
| `registerTool(tool)` | 注册工具 | AgentTool 实例 | void |
| `unregisterTool(name)` | 注销工具 | 工具名称字符串 | boolean |
| `getTool(name)` | 获取单个工具 | 工具名称 | AgentTool \| undefined |
| `getAllTools(options?)` | 获取所有工具(过滤后) | 策略选项 | AgentTool[] |
| `getToolDefinitions(options?)` | 获取LLM可用定义 | 策略选项 | any[] |

## 六、策略系统（权限控制）🛡️

### 6.1 为什么需要策略？

**问题场景**：
- 普通用户不应该使用 `write`（写入文件）或 `exec`（执行命令）
- 子Agent应该限制可用的工具
- 不同场景需要不同的工具集

**解决方案**：策略系统像门卫，决定谁能用什么工具

### 6.2 ToolPolicy（单级策略）

```typescript
interface ToolPolicyLike {
  allow?: string[];   // 允许的工具列表（白名单）
  deny?: string[];    // 拒绝的工具列表（黑名单）
}
```

**示例**：

```typescript
// 只允许读取文件
const readOnlyPolicy: ToolPolicyLike = {
  allow: ['read']
};

// 禁止执行命令
const safePolicy: ToolPolicyLike = {
  deny: ['exec', 'write']
};
```

### 6.3 策略管线（Pipeline）多级过滤 ⭐⭐

**类比**：像机场安检，需要过好几道检查

```
原始工具列表(10个)
    ↓
[第1关: Profile策略] 剩余8个
    ↓
[第2关: Global策略] 剩余7个
    ↓  
[第3关: Agent策略] 剩余6个
    ↓
[第4关: Group策略] 剩余5个
    ↓
[第5关: Subagent策略] 剩余4个 ← 最终结果
```

**代码实现**：

```typescript
import { applyToolPolicyPipeline, buildDefaultToolPolicyPipelineSteps } from './tool-policy-pipeline';

// 构建默认管线（5层过滤）
const steps = buildDefaultToolPolicyPipelineSteps({
  profilePolicy: { allow: ['read'] },      // 用户配置
  globalPolicy: { deny: ['exec'] },        // 全局配置
  agentPolicy: undefined,                   // 无Agent限制
  groupPolicy: undefined,                   // 无组限制
  subagentPolicy: { allow: ['read'] },     // 子Agent限制
});

// 应用管线
const filteredTools = applyToolPolicyPipeline({
  tools: allTools,
  steps: steps
});
```

### 6.4 工具组（批量管理）

可以用**别名**一次引用多个工具：

```typescript
import { TOOL_GROUPS, expandToolGroups } from './tool-groups';

// 预定义的组
TOOL_GROUPS = {
  fs: ['read', 'write', 'edit', 'apply_patch'],  // 文件系统组
  runtime: ['exec', 'process'],                    // 运行时组
  web: ['web_search', 'web_fetch'],                // Web组
};

// 展开组名为具体工具名
expandToolGroups(['fs', 'web_search']);
// → ['read', 'write', 'edit', 'apply_patch', 'web_search']
```

## 八、结果标准化

### 8.1 为什么需要标准化？

不同工具可能返回不同格式的结果，标准化让下游处理更简单：

```typescript
import { 
  normalizeToolExecutionResult,
  buildToolExecutionErrorResult,
  textResult,
  jsonResult
} from './result-normalizer';

// 标准化正常结果
normalizeToolExecutionResult({ toolName: 'read', result: rawResult });

// 构建错误结果（不抛异常！）
buildToolExecutionErrorResult({ 
  toolName: 'read', 
  message: '文件不存在' 
});
// → { content: [{ type: 'text', text: '{"status":"error",...}' }] }

// 快捷辅助函数
textResult('操作成功');           // 创建文本结果
jsonResult({ code: 200, data });  // 创建JSON结果
```

### 8.2 "错误转结果"策略 ⭐ 重要设计

传统做法：工具出错 → 抛异常 → 整个程序崩溃
本项目做法：工具出错 → 返回错误结果对象 → 继续运行

```typescript
// adapter.ts 中的关键代码
try {
  // ... 执行工具
  return normalizeToolExecutionResult(...);
} catch (err) {
  // ★ 错误不抛出，而是转成结果对象
  return buildToolExecutionErrorResult({
    toolName: tool.name,
    message: err.message
  });
}
```

**好处**：Agent不会因为一个工具失败而停止工作！

## 十、如何扩展：添加自定义工具

### 步骤1：定义工具

```typescript
// my-web-search-tool.ts
import { AgentTool, AgentToolResult } from './core';
import { z } from 'zod';

export function createWebSearchTool(): AgentTool<{ query: string }> {
  return {
    name: 'web_search',
    label: '网络搜索',
    description: '搜索互联网获取最新信息',
    
    // 用Zod定义参数（自动验证+生成JSON Schema）
    parameters: z.object({
      query: z.string().describe('搜索关键词'),
      count: z.number().optional().default(10)
    }),
    
    async execute(toolCallId, params, signal, onUpdate) {
      onUpdate?.(`正在搜索: ${params.query}`);
      
      // 调用搜索API...
      const results = await searchAPI(params.query, params.count);
      
      return {
        content: [{ type: 'text', text: JSON.stringify(results) }],
        details: { count: results.length }
      };
    }
  };
}
```

### 步骤2：注册到注册表

```typescript
import { ToolRegistry } from './tool-registry';
import { createWebSearchTool } from './my-web-search-tool';

const registry = new ToolRegistry();
registry.registerTool(createWebSearchTool());

// 现在AI就可以使用 web_search 了！
```

## 十二、常见问题FAQ

### Q1: Zod 是什么？为什么要用它？
**A**: Zod 是一个 TypeScript 类型验证库。我们用它来定义工具参数，好处是：
- 自动验证输入是否合法
- 自动生成 JSON Schema 给 LLM 看
- TypeScript 类型自动推断

### Q2: ownerOnly 是什么意思？
**A**: 标记为 `ownerOnly: true` 的工具只有"所有者"（管理员）可以使用。非所有者调用时会被策略过滤器移除。

### Q3: 如何禁用某个工具？
**A**: 在策略中使用 deny 列表：
```typescript
getAllTools({ agentPolicy: { deny: ['exec'] } })
```

### Q4: 工具执行超时怎么办？
**A**: 可以通过 `signal` 参数传递 `AbortSignal` 来取消长时间运行的工具：
```typescript
const controller = new AbortController();
setTimeout(() => controller.abort(), 5000);  // 5秒超时
tool.execute(id, params, controller.signal);
```
