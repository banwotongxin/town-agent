## 一、为什么需要记忆系统？

### 1.1 生活化类比

想象一下，如果一个角色每次和你聊天都像**失忆症**一样：

```
❌ 没有记忆系统的对话：
  你: "我叫小明"
  角色: "你好啊！很高兴认识你！"
  你: "我刚才说我叫什么？"
  角色: "呃...不好意思，我忘了..."

✅ 有记忆系统的对话：
  你: "我叫小明"
  角色: "你好啊小明！很高兴认识你！"  ← 记住了！
  你: "我刚才说我叫什么？"
  角色: "你叫小明呀，我们刚开始聊天的。"  ← 还记得！
```

### 1.2 记忆系统的三大作用

| 作用 | 说明 | 类比 |
|------|------|------|
| **上下文连续** | 让对话自然流畅 | 人能记得刚才说了什么 |
| **个性化体验** | 记住用户的偏好 | 朋友知道你喜欢什么 |
| **长期积累** | 从历史中学习成长 | 经验越丰富 |

## 二、三层记忆架构

### 2.1 全景图

```

│                    赛博小镇记忆架构                          │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Layer 1: 短期记忆（Short-Term Memory）                    │
│  ────────────────────────────────────────────────────────  │
│  📦 存储位置: 内存中                                        │
│  👁 保留时间: 最近 5 轮对话                                │
│  💾 类比: 电脑内存（RAM）- 断电就丢失                  │
│  🔧 用途: 当前对话上下文，确保回复连贯                       │
│                                                             │
│  Layer 2: 会话记忆（Session Memory）                     │
  ────────────────────────────────────────────────────────  │

│                                                             │
│  Layer 3: 长期记忆（Long-Term Memory / Knowledge Base）      │
  ────────────────────────────────────────────────────────  │
│  📦 存储位置: ChromaDB 向量数据库                        │
  👁 保留时间: 永久保存（跨会话）                             │
│  💾 类似: 图书馆 - 可以按语义搜索                           │
│  🔧 用途: 专业知识、重要事件、长期偏好                       │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 2.2 三层协作流程

```
用户发送: "我昨天问你关于Python的问题"

↓ Layer 1: 短期记忆查找
   检查最近5轮对话... 
   找到了！返回上下文

↓ Layer 2: 会话记忆补充
   加载本次会话的摘要...
   补充了背景信息

↓ Layer 3: 长期记忆检索
   在ChromaDB中搜索"Python"相关记忆...
   找到了之前学过的Python知识！

↓ 三层合并 → 发送给AI
   AI现在拥有了完整的上下文，
   可以给出精准的回复！
```

## 三、短期记忆（ShortTermMemory）

### 3.1 它是什么？

**短期记忆 = 最近几轮对话的缓存**

就像你的"工作记忆"——正在做的事情放在脑子里，但过一会儿就忘了细节。

### 3.2 核心属性

```typescript
// 文件位置：src/AI/memory/storage/dual_memory.ts

class ShortTermMemory {
  windowSize: number;        // 窗口大小（默认5轮）
  messages: BaseMessage[];  // 消息列表
  summary: string;           // 压缩后的旧对话摘要
  conversationRounds: number;// 对话轮数
  toolResultsLRU: Map;     // 工具结果缓存（LRU淘汰算法）
}
```

### 3.3 滑动窗口机制

```
窗口大小 = 5（默认）

[消息1] [消息2] [消息3] [消息4] [消息5] [消息6]新
 ↑___________________________________↑
 |           |                |
 已超出窗口！  |      当前窗口      |
             ↓                ↓
      压缩成摘要        保留原样
```

**当消息超过窗口大小时：**
1. 把最早的消息压缩成摘要
2. 只保留最近的 N 条消息
3. 这样既节省空间，又不丢失关键信息

### 3.4 工具结果的特殊处理

```
普通消息: [用户] [AI] [用户] [AI] [工具结果] [工具结果] [工具结果]
                                              ↑
                                        当工具结果过多时...
                                        
处理方式: 使用 LRU（最近最少使用）算法
- 只保留最近使用的 5 个工具结果
- 其他的替换为占位符 "[Old tool result cleared]"
- 避免工具结果占用太多空间
```

## 四、会话记忆（SessionMemory）

### 4.1 它是什么？

**会话记忆 = 一次完整对话的摘要笔记**

就像开会时的**会议纪要**——把长对话浓缩成要点。

### 4.2 何时触发记忆提取？

```typescript
// 文件位置：src/AI/memory/storage/session_memory.ts

shouldExtractMemory(messages, tokenCount): boolean {
  // 条件1: token 数超过 5000（对话足够长）
  // 条件2: 距上次提取超过 2000 token
  // 条件3: 至少有 3 次工具调用
  
  // 所有条件满足才触发提取
  return (tokenCount >= 5000 && 
          tokensSinceLastExtraction >= 2000 && 
          toolCallCount >= 3);
}
```

### 4.3 摘要文件结构

会话记忆保存在 `memory_storage/roles/{agent_id}_summary.md`：

```markdown
# Session Title
# Current State
# Task specification
# Files and Functions
# Workflow
# Errors & Corrections
# Codebase and System Documentation
# Learnings
# Key results
# Worklog

[Updated at 2026-04-12T15:30:00]

用户之前问了关于Python的问题，讨论了列表推导式、装饰器模式...
```

### 4.4 会话压缩功能

当对话太长需要压缩时：

```
原始消息（15000 tokens）:
  [消息1][消息2]...[消息100]

                    ↓ compactSessionMemory()

压缩后:
  [边界标记] + [会话摘要] + [最近N条消息]
  
  大约 5000-8000 tokens！
  大幅减少，但保留了关键信息
```

## 五、长期记忆/知识库（Long-Term Memory）

### 5.1 它是什么？

**长期记忆 = 角色的"图书馆"**

用**向量数据库（ChromaDB）**存储，支持按**语义相似度**搜索！

### 5.2 为什么用向量数据库？

传统数据库 vs 向量数据库对比：

```
传统数据库（MySQL等）:
  查询: SELECT * FROM memories WHERE content LIKE '%Python%'
  问题: 只能精确匹配，搜不到"编程语言"、"代码"等相关内容

向量数据库（ChromaDB）:
  查询: 找出与 "Python" 语义最接近的记忆
  结果: 能找到 "学习Python"、"Python编程"、"代码例子" 等！
  因为它理解"意思相近"！
```

### 5.3 ChromaDB 工作原理

```
存储过程:
  1. "我喜欢用Python做数据分析" 这段文字
           ↓
  2. Embedding模型把它变成向量 [0.12, -0.34, 0.56, ...]
     （一堆数字，代表这段文字的"语义指纹"）
           ↓
  3. 存入 ChromaDB，关联原文和向量

查询过程:
  1. 用户问 "我想学编程"
           ↓
  2. Embedding模型也把问题变成向量
           ↓
  3. 在数据库中找"语义接近"的向量
     （计算两个向量的相似度）
           ↓
  4. 返回最相关的记忆！
```

### 5.4 ChromaLongTermMemory 实现

```typescript
// 文件位置：src/AI/memory/storage/chroma_long_term_memory.ts

class ChromaLongTermMemory implements LongTermMemoryInterface {
  
  // 添加记忆
  async addMemory(content, importance = 0.5, metadata?) {
    // 1. 文本分块（太长的文本拆成小块）
    const chunks = this.chunker.chunk(content);
    
    // 2. 向量化（文本转向量）
    const embeddings = await this.embedder.embed(chunks);
    
    // 3. 存入ChromaDB
    await this.collection.add({
      ids: chunks.map((_, i) => `id_${Date.now()}_${i}`),
      documents: chunks,      // 原始文本
      embeddings: embeddings,  // 向量
      metadatas: { importance, timestamp, ...metadata }
    });
  }
  
  // 搜索记忆（语义搜索）
  async search(query, topK = 3, minImportance = 0.0) {
    // 1. 问题改写（扩展同义词和相关表达）
    const rewrittenQueries = this.rewriter.rewriteQuestion(query);
    
    // 2. 向量化查询
    const queryEmbeddings = this.embedder.embed(rewrittenQueries);
    
    // 3. 在ChromaDB中进行向量相似度搜索
    for (const embedding of queryEmbeddings) {
      const results = await this.collection.query({
        queryEmbeddings: [embedding],
        nResults: topK
      });
      // 处理结果...
    }
    
    // 4. 结果精排（重新排序确保最相关的排前面）
    return this.reranker.rerank(query, uniqueResults, topK);
  }
}
```

### 5.5 记忆的重要性评分

每条记忆都有一个重要性评分（0-1）：


## 六、双层记忆协调器（DualMemorySystem）

### 6.1 它是什么？

**DualMemorySystem = 记忆系统的"总指挥官"**

统一管理短期、会话、长期三层记忆，对外提供简单的接口。

### 6.2 核心接口

```typescript
// 文件位置：src/AI/memory/storage/dual_memory.ts

class DualMemorySystem {
  private shortTerm: ShortTermMemory;    // 短期记忆
  private longTerm: LongTermMemoryInterface; // 长期记忆（知识库）
  private sessionMemory: SessionMemory;     // 会话记忆
  
  // 添加消息（自动分发到合适的层级）
  async addMessage(message: BaseMessage) {
    // 1. 添加到短期记忆
    this.shortTerm.addMessage(message);
    
    // 2. 检查是否需要提取会话记忆
    if (this.sessionMemory.shouldExtractMemory(messages, tokenCount)) {
      await this.sessionMemory.extractMemory(messages);  // 提取会话摘要
    }
  }
  
  // 获取上下文（整合三层记忆）
  async getContext(query?: string): Promise<string> {
    const parts: string[] = [];
    
    // 1. 短期记忆（当前窗口内的对话）
    parts.push(this.shortTerm.getContext());
    
    // 2. 会话记忆（历史对话摘要）
    const sessionMem = this.sessionMemory.getSessionMemory();
    if (sessionMem) {
      parts.push(`[会话记忆]\n${sessionMem}`);
    }
    
    // 3. 长期记忆/知识库（如果提供了查询词）
    if (query) {
      const relevantMemories = await this.longTerm.search(query, 3);
      if (relevantMemories.length > 0) {
        parts.push(`[相关知识库]\n${memoriesText}`);
      }
    }
    
    return parts.join('\n\n');
  }
}
```

## 八、Token 管理

### 8.1 什么是 Token？

**Token = AI 处理文字的基本单位**

大致换算：
- 1 个汉字 ≈ 1-2 tokens
- 1 个英文单词 ≈ 1-2 tokens
- 1 行代码 ≈ 30-50 tokens

### 8.2 各层 Token 限制

| 层级 | 默认限制 | 说明 |
|------|---------|------|
| 短期记忆窗口 | 5000 tokens | 约2500-5000汉字 |
| 会话记忆提取阈值 | 5000 tokens | 触发提取 |
| 单个摘要最大 | 12000 tokens | 一次摘要的上限 |
| LLM 上下文预算 | 8000 tokens | 发送给AI的最大量 |

### 8.3 Token 计算工具

```typescript
// 文件位置：src/AI/memory/compression/token_utils.ts

class TokenUtils {
  // 计算单条消息的token数
  static calculateTokenCount(text: string): number {
    // 使用 tiktoken 库精确计算
    // 或者简单估算：中文字符数 * 1.5 + 英文单词数 * 1.3
  }
  
  // 计算整个消息列表的token总数
  static calculateMessagesTokenCount(messages: BaseMessage[]): number {
    let total = 0;
    for (const msg of messages) {
      total += this.calculateTokenCount(msg.content);
    }
    return total;
  }
}
```

## 九、记忆压缩策略

### 9.1 什么时候需要压缩？

```
检测条件（满足任一即触发）：
├── 短期记忆 > 7500 tokens
├── 对话轮数 > 10 轮
└── 工具结果占用太多空间
```

### 9.2 压缩算法选择

```
预防性压缩路由决策:

情况1: 只是轻微超限（< 20%）
  → 只截断工具结果，保留对话原文
  → 操作快，信息损失小

情况2: 中度超限（20%-50%）
  → 先截断工具结果，再进行主动压缩
  → 用LLM生成摘要替代旧消息

情况3: 严重超限（> 50%）
  → 全面主动压缩
  → 保留最重要的信息，大幅缩减体积
```

### 9.3 主动压缩过程

```
原始: 50条消息，15000 tokens

Step 1: 选择保留边界
  → 保留最近5条消息不动
        
Step 2: 对边界前的消息调用LLM生成摘要
  Input:  [消息1...消息45]
  Output: "用户询问了Python问题，讨论了pandas和numpy..."
        
Step 3: 构建压缩后消息
  [摘要消息] + [消息46...消息50]
        
Result: ~6000 tokens！节省60%
```

## 十、常见问题

### Q1: 记忆占用了太多磁盘空间？
**解决方法**:
- 定期清理旧的 `history_backup.json` 文件
- 调整 `windowSize` 减少短期记忆保留量
- 清理 ChromaDB 中低重要性的记忆

### Q2: 角色记住了错误的记忆？
**原因**: 长期记忆中存入了噪声数据
**解决方法**:
- 调整 `minImportance` 阈值，过滤低质量记忆
- 手动清理 `chroma/` 目录

### Q3: 对话变慢了？
**原因**: 记忆太多，每次都要搜索
**优化方案**:
- 减少长期记忆搜索频率
- 降低 `topK` 参数（默认只搜3条）
- 启用缓存机制

### Q4: 如何导出/导入记忆？
**当前支持**:
- 对话历史：直接复制 `memory_storage/roles/` 目录
- 向量记忆：导出 ChromaDB 的 collection
